    A Real Class Act
    When and why to use classes, instance variables, and instance methods
    October 19, 2014

    Phew! The use of classes along with instance variables and methods has been somewhat confusing this week. While I understand the high-level concept of why and when they are used, I am still trying to grasp what can and can't be done within (and without) them.

    I do know that classes in Ruby is what makes the language so powerful, though. Hopefully by the end of this post, I will have a more solid understanding of why. I'll push forward with using cars as an example class to help better understand the concept (for you and me both). Here goes nothing!

    1 class Cars
    2    def initialize(make, model, year)
    3     @make = make
    4     @model = model
    5     @year = year
    6    end
    7
    8    def display_car_stats
    9      "I am a #{@make} #{@model} made in #{@year}!"
    10   end
    11 end

    Initialize (line 2) is a special method in Ruby programs. When you call Cars.new to create a new Cars object, Ruby creates an uninitialized object and then calls that object's initialize method, passing in any parameters that were passed to new. The parameters we have identified are make, model, and year (line 2). This gives you a chance to write code that sets up your object's state (this explanation has been modified and derived from source).

    The "@" sign in lines 3-5 above is used to store local variables as instance variables. This will allow the variables to be accessed outside of the various methods in a class. This also implies that instance variables change from object to object.

    So why would we need a class of cars, as demonstrated by the example above? Each car will have a set of characteristics. For example, we used 3 parameters - make, model, and year - to help us identify each car. Of course, there could be many more characteristics (i.e. car color, 0-60mph rate, etc.), but we want to keep it simple for the sake of this example. The main takeaway here is that it each new object (car) will have these declared characteristics that will be stored and will help us identify them for later use, like in instance methods. It also saves us a lot of time and energy, especially if we have hundreds of cars with different characteristics to list.

    Let's say we want to add a new car with its characteristics in our class. This is how it looks:


    Cars.new("Subaru", "Impreza", "2011")

    Here we've created a new object, in our case a car. By defining our Cars class, we can group behaviors and characteristics that our cars have. We can also use the methods within the class and apply it to this car or any other car that has been created.

    So let's use the display_car_stats method from our class (line 8)! Here's what happens once we've inputted the new Car object that I have named mysubie:


    mysubie = Cars.new("Subaru", "Impreza", "2011")

    mysubie.display_car_stats

    # => "I am a Subaru Impreza made in 2011!"

    Note that in our class code above, we call out our instance variables within the Cars#display_car_stats method with the following syntax: #{@make} #{@model} #{@year}. Another sidenote to remember is that we can identify Class methods with the following syntax (which I use at the beginning of this paragraph: Class#method_name.

    Woo! I just tested it all in IRB and it works just fine. I'm hoping to do more within classes, i.e. modifying objects and applying other methods for only certain objects (and not all). Perhaps there will be a Part 2 of this post soon!
